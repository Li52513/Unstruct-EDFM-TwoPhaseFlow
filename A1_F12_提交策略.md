## 2. 总体原则
- 不按 `M1-S1`、`M1-S2` 这种单个步骤机械提交。
- 按“最小可验证闭环”提交。
- 一次提交只做一类事情，不混合多个主题。
- 只有当改动达到“主题单一 + 状态自洽 + 可验证”三条件时，才建议提交。

## 3. 三条件判断法
### 3.1 主题单一
- 本次改动只围绕一个主题。
- 合格示例：
  - 契约层字段冻结
  - `A1` 解析验证链抽离
  - `B1` reference I/O 抽离
  - `A7` 的 `N=1 + wells` 打通
- 不合格示例：
  - 同一提交里既改公共模块，又改多个无关工况模板，又改文档规则

### 3.2 状态自洽
- 当前改动是一个完整的小闭环，而不是半拉子中间态。
- 必须满足：
  - 代码或文档逻辑前后一致
  - 当前工作区可以停下来，不会让后续人接手时摸不清状态
  - 不依赖“下一步再补一点”才能成立

### 3.3 可验证
- 本次改动至少有一种明确验证依据。
- 验证可以是：
  - 编译通过
  - `--list` / `--case` / `--stage` 入口可运行
  - 对应工况或脚本可执行
  - 文档一致性检查通过
  - 验证图件/脚本/报告产物已经生成

## 4. 每步完成后的固定判断流程
每完成一个步骤后，固定按下面顺序判断：

1. 看该步骤是否只产生了“文档草稿”或“代码半成品”。
- 如果是，先不提交。

2. 看该步骤是否已经形成“最小可验证闭环”。
- 如果没有，先不提交。

3. 看当前 `git status` 中是否混入无关文件。
- 如果混入无关文件，先清理或明确排除，再决定是否提交。

4. 看是否满足“三条件判断法”。
- 三项都满足：建议提交。
- 任一不满足：不提交，继续推进。

5. 若建议提交，先告知用户以下信息：
- 为什么现在是合适提交点
- 本次准备提交哪些文件
- 哪些文件不会提交
- 建议提交信息

## 10. 告知用户的固定格式
当判断“现在应该提交”时，固定向用户报告：

- `提交建议：是`
- `原因：` 为什么当前已形成最小可验证闭环
- `建议提交文件：` 本次应纳入提交的文件
- `建议排除文件：` 本次不应提交的文件
- `建议提交信息：` 一条清晰的 commit message

当判断“现在不该提交”时，固定向用户报告：

- `提交建议：否`
- `原因：` 当前仍是中间态/半闭环
- `下一合适提交点：` 哪个步骤或哪组步骤完成后再提交
